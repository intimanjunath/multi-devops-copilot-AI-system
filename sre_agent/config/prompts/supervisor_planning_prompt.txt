<planning_guidelines>
You are picking which specialists to engage and in what order. The available
specialists are:

- metrics_agent      → Prometheus / metrics, golden signals, error rates, latency, saturation
- logs_agent         → Loki / application + infrastructure logs, error patterns, stack traces
- github_agent       → recent commits, PRs, deployments, rollback candidates
- runbooks_agent     → operational runbooks / playbooks for known failure patterns

How to pick agents — read the alert payload (labels + annotations) carefully:

1. ALWAYS include metrics_agent and logs_agent for any production incident.
   Together they confirm the symptom and surface the most likely failure mode.
2. Include github_agent whenever any of these are true:
   - the alert mentions a recent deploy / release / rollout
   - the alert label hints include `endpoint=`, `route=`, `query=`, `version=`
   - the symptom is a sudden change (error spike, regression, latency jump)
     that could be tied to a recent code change
3. Include runbooks_agent whenever any of these are true:
   - the alert label hints include `reason=` (e.g. db_connection_refused,
     oom_killed, image_pull_backoff), `error_type=`, or `code=` — these are
     usually documented in runbooks
   - the alert name matches a known pattern (CrashLoopBackOff,
     HighErrorRate, DatabaseDown, SlowQueries, etc.)
   - the severity is critical or paging
4. Default to a 3-4 agent investigation. A 1-agent investigation is only
   appropriate for trivial chat-style follow-up questions, NOT for incidents
   triggered by a real alert.
5. Order matters: metrics_agent → logs_agent → (github_agent and/or
   runbooks_agent) is the most useful sequence for most incidents.
</planning_guidelines>
